# Web Interface Guidelines

Цей документ містить суворі стандарти розробки інтерфейсів, забезпечення доступності (a11y), роботи з типографікою, формами, анімаціями та локалізацією. Кожен компонент інтерфейсу має відповідати цим вимогам перед інтеграцією в додаток.

---

## 1. Accessibility (Доступність)

*   **Кнопки-іконки:** Будь-яка кнопка, яка містить лише іконку без тексту, зобов'язана мати пропс `aria-label` для екранних зчитувачів.
    *   ❌ Bad: `<button><svg>...</svg></button>`
    *   ✅ Good: `<button aria-label="Close dialog"><svg>...</svg></button>`
*   **Елементи форм:** Усі текстові поля, випадаючі списки та чекбокси повинні мати зв'язаний тег `<label>` (через `htmlFor`) або явний атрибут `aria-label`/`aria-labelledby`.
*   **Інтерактивні не-HTML елементи:** Якщо ви змушені вішати події кліку на не-кнопки (наприклад, інтерактивний SVG-графік), обов'язково додавайте обробники клавіатури (`onKeyDown` або `onKeyUp`) та атрибут `tabIndex={0}`.
*   **Семантичний вибір тегів:** Використовуйте нативний `<button>` для дій на сторінці (відкриття модалок, видалення, збереження) та тег `<a>` (або компонент `<Link>`) для навігації між сторінками. **Категорично заборонено** вішати навігаційні кліки на `<div onClick>`.
*   **Зображення:** Кожен тег `<img>` повинен мати атрибут `alt`. Якщо зображення є суто декоративним (ілюстрація на фоні), передавайте порожній рядок `alt=""`.
*   **Декоративні іконки:** Якщо іконка дублює текст поруч, приховуйте її від скрінрідерів через `aria-hidden="true"`.
    *   ✅ Good: `<button aria-label="Delete message"><svg aria-hidden="true">...</svg> Delete</button>`
*   **Асинхронні сповіщення:** Тости, банери валідації та асинхронні повідомлення про статус завантаження повинні мати атрибут `aria-live="polite"` або `aria-live="assertive"`, щоб користувачі скрінрідерів знали про зміни на екрані.
*   **Структура заголовків:** Дотримуйтесь строгої ієрархії тегів заголовків від `<h1>` до `<h6>`. Не використовуйте теги заголовків лише для того, щоб зробити текст більшим — змінюйте розмір через класи CSS.
*   **Швидкий перехід:** Додавайте невидиму посилання-пропуск (`Skip to main content`) для користувачів, які навігують за допомогою клавіатури.
*   **Якірні посилання:** Для заголовків з анкорами (якірними посиланнями) додавайте CSS властивість `scroll-margin-top`, щоб при переході за посиланням заголовок не ховався під фіксований хедер сторінки.

## 2. Focus States (Стани фокусу)

*   **Видимість фокусу:** Усі інтерактивні елементи повинні мати чітко помітний фокус при навігації клавіатурою. Використовуйте для цього утилітарні класи Tailwind типу `focus-visible:ring-*` або аналогічні CSS-стилі.
*   **Заборона повного вимкнення фокусу:** Категорично заборонено прописувати стилі `outline: none` або `outline-none` без негайної заміни їх на кастомний, помітний ефект фокусу.
*   **Покращений селектор фокусу:** Надавайте перевагу псевдокласу `:focus-visible` перед стандартним `:focus`. Це запобігає появі чорної рамки фокусу при звичайному кліку мишею (що часто дратує користувачів), залишаючи її лише при навігації клавіатурою.
*   **Груповий фокус:** Для складних інтерактивних блоків (наприклад, кастомні картки-кнопки) використовуйте псевдоклас `:focus-within` на батьківському елементі, щоб підсвічувати всю картку, коли фокус знаходиться на внутрішніх інпутах чи кнопках.

## 3. Forms (Робота з формами)

*   **Автозаповнення:** Кожне текстове поле повинно мати осмислений атрибут `name` та коректний атрибут `autocomplete` (наприклад, `autocomplete="email"` або `autocomplete="current-password"`), щоб допомогти браузерам та менеджерам паролів.
*   **Мобільний ввід:** Використовуйте точні атрибути `type` (`email`, `tel`, `url`, `number`) та `inputmode` (наприклад, `inputmode="decimal"` для дрібних чисел або `inputmode="numeric"` для кодів підтвердження). Це автоматично відкриває правильну розкладку клавіатури на смартфонах.
*   **Заборона блокування вставки:** Ніколи не блокуйте вставку тексту в інпути (не перехоплюйте `onPaste` для виклику `preventDefault()`). Це повністю руйнує роботу з паролями з менеджерів безпеки та копіювання довгих даних.
*   **Кліки по лейблах:** Переконайтеся, що натискання на текст лейбла автоматично ставить фокус у зв'язаний інпут.
*   **Спелчекер:** Вимикайте автоперевірку орфографії для полів введення email, кодів валідації, токенів та юзернеймів за допомогою атрибуту `spellCheck={false}`.
*   **Область кліку чекбоксів:** Текст лейбла та сам інпут чекбокса/радіо-кнопки повинні ділити **єдину спільну та велику область кліку**. Користувач не повинен цілитися у крихітний квадратик чекбокса.
*   **Стан відправки форми:** Кнопка Submit має залишатися активною під час заповнення та ставати неактивною тільки після безпосереднього кліку під час виконання запиту. Показуйте лоадер (Spinner) або анімацію завантаження всередині кнопки.
*   **Валідація та помилки:** Виводьте повідомлення про помилки валідації безпосередньо біля проблемного поля. Після спроби відправки форми з помилками автоматично переводьте фокус на перше поле з помилкою.
*   **Красиві плейсхолдери:** Текст плейсхолдерів в інпутах повинен закінчуватися трикрапкою `…` та демонструвати реальний патерн вводу (наприклад, `Placeholder: "ivanov@domain.com…"`).
*   **Приховування від менеджерів паролів:** Не пишіть `autocomplete="off"` на звичайних текстових полях, якщо ви просто хочете вимкнути історію вводу — це часто ігнорується. Для чутливих полів використовуйте спеціальні значення (наприклад, `autocomplete="new-password"` для полів реєстрації).
*   **Попередження про втрату даних:** Якщо користувач вніс зміни у форму і намагається закрити вкладку або перейти на іншу сторінку, показуйте вікно попередження про втрату незбережених даних (`beforeunload` подія або блокування роутера).

## 4. Animation (Анімація та переходи)

*   **Турбота про здоров'я:** Завжди поважайте системні налаштування користувача щодо обмеження руху. Обов'язково додавайте медіазапит `prefers-reduced-motion` для повного вимкнення або значного сповільнення та спрощення анімацій.
*   **Оптимальні властивості для GPU:** Анімуйте виключно властивості, які підтримуються композитором відеокарти без запуску повного reflow/repaint сторінки — це `transform` (scale, translate, rotate) та `opacity`.
*   **Заборона універсальних транзицій:** **Категорично заборонено** використовувати властивість `transition: all`. Завжди явно перераховуйте властивості, які переходять у часі.
    *   ❌ Bad: `transition: all 0.3s ease;`
    *   ✅ Good: `transition: transform 0.2s ease, opacity 0.2s ease;`
*   **Опорна точка трансформації:** Завжди явно вказуйте властивість `transform-origin` для анімацій масштабування або обертання, щоб уникнути хаотичного зсуву елементів у просторі.
*   **Трансформації в SVG:** Для надійного рендерингу анімацій всередині SVG вішайте класи трансформацій на груповий тег `<g>`, а також додавайте стилі `transform-box: fill-box; transform-origin: center;` у CSS.
*   **Переривання анімацій:** Усі анімації повинні бути перериваними. Якщо користувач ініціює нову дію посеред програвання анімації, інтерфейс має миттєво зреагувати на ввід, а не чекати завершення таймерів CSS.

## 5. Typography (Типографіка)

*   **Правильна трикрапка:** Завжди використовуйте спеціальний нативний символ трикрапки `…` (Alt+0133 на Windows або Option+; на Mac) замість трьох окремих крапок `...`.
*   **Кавичкі-ялинки:** Використовуйте красиві типографічні кавички (`«` та `»` або `“` та `”`) замість прямих програмістських кавичок (`"` або `'`).
*   **Нерозривні пробіли:** Використовуйте нерозривний пробіл `&nbsp;` (або `\u00A0` у JS) для запобігання відриву одиниць виміру, гарячих клавіш або сполучників.
    *   ✅ Good: `10&nbsp;MB`, `⌘&nbsp;K`, `Vite&nbsp;JS`.
*   **Статус завантаження:** Текст процесів, що тривають у часі, повинен закінчуватися трикрапкою та формулюватися у теперішньому тривалому часі.
    *   ✅ Good: `Loading…`, `Saving…`, `Publishing…`.
*   **Табличні цифри:** Для числових колонок у таблицях, лічильників, таймерів та фінансових звітів завжди застосовуйте CSS-стиль `font-variant-numeric: tabular-nums`. Це робить ширину кожної цифри однаковою, запобігаючи візуальному дрижанню тексту при швидкій зміні значень.
*   **Захист від висячих рядків:** Застосовуйте властивості `text-wrap: balance` або `text-pretty` для заголовків та коротких промо-абзаців. Це автоматично балансує довжину рядків, запобігаючи появі одного самотнього слова на новому рядку.

---

## 6. Content Handling (Обробка контенту)

*   **Захист від переповнення:** Будь-який контейнер, що відображає динамічний текст від користувача або з API, повинен мати захист від ламання верстки. Застосовуйте класи обмеження рядків `truncate`, `line-clamp-*` або примусове перенесення довгих слів `break-words`.
*   **Проблема Flex-елементів:** Дочірні елементи флекс-контейнерів (`flex-children`) ігнорують переповнення тексту всередині себе. Завжди явно задавайте їм властивість `min-w-0` (або `min-h-0`), щоб увімкнути механізм обмеження тексту (`truncate` чи `line-clamp`).
*   **Порожні стани (Empty States):** Завжди проектуйте гарні порожні стани. Якщо з сервера прийшов порожній масив або рядок, додаток не повинен показувати порожній білий екран чи зламану таблицю — виводьте інформативну ілюстрацію із кнопкою дії.
*   **Екстремальні дані:** Завжди тестуйте компоненти на трьох типах контенту: мінімальний (1 символ), середній (стандартний макет) та екстремально великий (текст на 5000 слів або суцільний набір літер без пробілів).

---

## 7. Images (Зображення)

*   **Запобігання стрибкам верстки (CLS):** Кожен тег `<img>` зобов'язаний мати явні атрибути `width` та `height` (або аналогічні розміри в CSS). Це допомагає браузеру зарезервувати правильне співвідношення сторін (Aspect Ratio) до моменту повного завантаження файлу зображення, повністю усуваючи зсув макету.
*   **Ліниве завантаження:** Для всіх зображень, які знаходяться нижче першого екрана (below the fold), завжди прописуйте атрибут `loading="lazy"`.
*   **Пріоритетне завантаження:** Для критично важливих зображень на першому екрані (банери, логотипи, аватари профілю) додавайте атрибут пріоритету `fetchpriority="high"`.

## 8. Performance (Продуктивність UI)

*   **Віртуалізація списків:** Якщо список або таблиця відображає понад 50-100 елементів одночасно, використовуйте бібліотеки віртуалізації (наприклад, `@tanstack/react-virtual`) або CSS-властивість `content-visibility: auto`.
*   **Заборона читання геометрії в рендері:** Ніколи не викликайте методи зчитування розмірів (`getBoundingClientRect`, `offsetHeight`, `scrollTop`) безпосередньо всередині функції рендерингу компонента — це моментально ламає продуктивність через постійні reflow. Робіть це лише в ефектах (`useEffect`/`useLayoutEffect`) або обробниках подій.
*   **Контрольовані інпути:** Надавайте перевагу неконтрольованим інпутам для великих форм. Якщо інпут контрольований (`value` + `onChange`), обробник оновлення стану повинен виконуватись миттєво без запуску важких супутніх обчислень на кожне натискання клавіші.
*   **Попереднє підключення до CDNs:** Для сторонніх доменів з картинками або шрифтами додавайте теги передз'єднання у `index.html`: `<link rel="preconnect" href="https://cdn.example.com">`.

---

## 9. Touch & Interaction (Сенсорна взаємодія)

*   **Усунення затримки кліку:** Для мобільних браузерів застосовуйте стиль `touch-action: manipulation`, який повністю вимикає затримку у 300мс, призначену для розпізнавання подвійного тапу для зуму.
*   **Кастомне підсвічування дотику:** Контролюйте колір підсвічування інтерактивних елементів при тапі на смартфонах за допомогою властивості `-webkit-tap-highlight-color`.
*   **Фіксація скролу підкладки:** Для відкритих модальних вікон, бічних штор (drawers) або випадаючих меню завжди задавайте властивість `overscroll-behavior: contain` на контейнері, щоб прокручування всередині модалки не прокручувало основну сторінку під нею.
*   **Стан перетягування (Drag & Drop):** Під час активного перетягування елементів повністю вимикайте виділення тексту на сторінці через CSS `user-select: none`, а також вішайте атрибут `inert` на фонові елементи, які не беруть участі у взаємодії.

---

## 10. Dark Mode & Theming (Темна тема та стилізація)

*   **Синхронізація системних елементів:** При активації темної теми обов'язково прописуйте властивість `color-scheme: dark` на тег `<html>`. Це автоматично змінює колір системного скролбару браузера, стандартних полів вводу та кнопок під темну палітру.
*   **Колір статус-бару:** Оновлюйте колір статус-бару на мобільних пристроях відповідно до поточної теми за допомогою мета-тегу `<meta name="theme-color" content="#16171d">`.

---

## 11. Locale & i18n (Локалізація та формати)

*   **Форматування дат:** Ніколи не хардкодьте відображення дат та часу на кшталт `DD.MM.YYYY`. Завжди використовуйте вбудований системний інструмент `Intl.DateTimeFormat` або перевірені локалізовані бібліотеки.
*   **Форматування фінансів та чисел:** Для відображення валют, відсотків та великих чисел використовуйте `Intl.NumberFormat`. Це гарантує правильний роздільник тисячних частин залежно від мови користувача (наприклад, пробіл в Україні, кома в США).
*   **Захист від автоперекладачів:** Для назв брендів, кодів помилок, технічних токенів та ідентифікаторів прописуйте атрибут `translate="no"`. Це захищає текст від некоректного або смішного перекладу автоматичними плагінами браузерів (наприклад, Google Translate).

---

## 12. Hydration Safety (Безпека гідратації)

*   **Контроль вводу:** Будь-яке поле вводу, що має пропс `value`, зобов'язане мати обробник `onChange`. Якщо обробник не потрібен (статичне значення за замовчуванням), використовуйте `defaultValue` замість `value`.
*   **Різниця серверного часу:** Оскільки рендеринг може відбуватися на сервері або у розробника на машині з іншим часовим поясом, ніж у клієнта, будь-яке відображення поточного часу або дат може призвести до помилки гідратації (Hydration Mismatch). Переносьте рендеринг дат у клієнтський життєвий цикл (`useEffect`) або використовуйте `suppressHydrationWarning` виключно для динамічних елементів дати, де різниця є очікуваною.

---

## 13. Copywriting & Tone (Копірайтинг та тексти)

*   **Активний стан:** Завжди використовуйте активний стан дієслів замість пасивного.
    *   ❌ Bad: "The API key will be generated after clicking the button."
    *   ✅ Good: "Click the button to generate your API key."
*   **Конкретні кнопки:** Текст на кнопках дій має бути максимально точним і відповідати результату кліку. Уникайте абстрактних слів.
    *   ❌ Bad: `Continue`, `OK`, `Proceed`.
    *   ✅ Good: `Save API Key`, `Delete Message`, `Create Invoice`.
*   **Конструктивні помилки:** Будь-яке повідомлення про помилку повинно складатися з двох частин: опис того, що сталося, та **чітка інструкція, як користувачу вирішити цю проблему**.
    *   ❌ Bad: "Invalid token format."
    *   ✅ Good: "Invalid token format. Please copy the token from your dashboard and try again."
*   **Амперсанд:** Використовуйте символ `&` замість слова "and" виключно у тісних елементах інтерфейсу, де є жорстке обмеження простору (таблиці, бейджі). У звичайних текстах пишіть повноцінне слово.

## 14. Anti-patterns (Головні антипатерни)

AI-агент повинен негайно сигналізувати про виявлення наступних конструкцій та пропонувати їх рефакторинг:
1. Вимкнення зуму сторінки на мобільних: `user-scalable=no` або `maximum-scale=1` у мета-тегу viewport.
2. Блокування вставки через `preventDefault()` в події `onPaste`.
3. Використання `transition: all`.
4. Застосування `outline-none` або `outline: none` на кнопках чи інпутах без налаштування кастомного `:focus-visible`.
5. Навігаційні кліки на не-семантичних елементах (наприклад, `<div onClick={navigate}>`).
6. Текстові інпути форм без зв'язаних лейблів або атрибутів `aria-label`.
7. Зображення `<img>` без вказаних розмірів `width`/`height` або `alt`.
8. Наявність жорстко прописаних шаблонів форматування дат (на кшталт `const date = day + "." + month`).
